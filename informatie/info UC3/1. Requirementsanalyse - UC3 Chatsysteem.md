## 1. Inleiding

### 1.1 Doel
Dit document beschrijft alle functionele en niet-functionele eisen voor UC3 – het chatsysteem van de Showcase-applicatie. Het dient als basis voor het functioneel en technisch ontwerp.

### 1.2 Aanleiding
De opdrachtgever (school/docenten) vereist een showcase-portfolio-applicatie waarmee de student aantoont competenties te bezitten in:
- Authenticatie, autorisatie en beveiliging (US4)
- Realtime frontend-integratie (US5)
- Testen en deployment (US6)

UC3 is het chatsysteem dat deze drie user stories samenbrengt.

### 1.3 Context
UC3 bouwt voort op:
- **UC1** – Profielpagina (statische MVC-pagina, Bootstrap, CV)
- **UC2** – Contactpagina (formulier, REST API, Mailtrap e-mail)

UC3 voegt authenticatie, rollen, 2FA, realtime communicatie en uitgebreide testdekking toe.

## 3. User Stories

De eisen zijn afgeleid van de officiële Studentenhandleiding Showcase (US4–US6).

### US4 – Authenticatie en Autorisatie
> _Als gebruiker wil ik mij kunnen registreren en inloggen, zodat ik veilig toegang heb tot de chat._

> _Als beheerder wil ik speciale bevoegdheden hebben, zodat ik het systeem kan beheren._

> _Als gebruiker wil ik Two-Factor Authentication kunnen activeren, zodat mijn account extra beveiligd is._

### US5 – Realtime Communicatie
> _Als ingelogde gebruiker wil ik realtime berichten kunnen sturen en ontvangen, zodat ik direct kan communiceren met andere gebruikers._

> _Als gebruiker wil ik een moderne Vue-interface zien, zodat de chatervaring vloeiend en responsief is._

### US6 – Kwaliteit, Testen en Deployment
> _Als ontwikkelaar wil ik dat de applicatie grondig getest is (unit + E2E), zodat de kwaliteit geborgd is._

> _Als gebruiker wil ik dat de applicatie veilig is (XSS, SQLi, CSRF), zodat mijn data beschermd is._

## 8. Aannames

1. Eén gedeelde chatroom voor alle gebruikers (geen rooms/kanalen)
2. Berichten zijn zichtbaar voor alle ingelogde gebruikers (geen privacy-scope)
3. De applicatie draait lokaal (SQLite) en hoeft niet horizontaal te schalen
4. TOTP-synchronisatie wordt verwacht van de gebruiker (juiste kloktijd)
5. Admin-account wordt aangemaakt via seeding; geen registratie-UI voor admin
6. Berichtenhistorie is max. 100 berichten (nieuwste eerst) voor performantie

## 10. Begrippenlijst

| Begriff | Definitie |
|---------|-----------|
| TOTP | Time-based One-Time Password (RFC 6238) – 6-cijferige code gegenereerd door authenticator-app |
| SignalR | ASP.NET Core bibliotheek voor realtime bi-directionele communicatie via WebSocket/long-polling |
| Hub | SignalR server-component; clients verbinden met een hub om berichten te sturen/ontvangen |
| ASVS | Application Security Verification Standard van OWASP |
| JWT | JSON Web Token – niet gebruikt in UC3 (cookie-authenticatie is gekozen) |
| EF Core | Entity Framework Core – ORM voor .NET |
| CSP | Content Security Policy – HTTP-header die bronnen van scripts/stijlen beperkt |
| HSTS | HTTP Strict Transport Security – dwingt HTTPS af via HTTP-header |
| MoSCoW | Prioriteringstechniek: Must/Should/Could/Won't |

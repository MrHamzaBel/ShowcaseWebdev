## 1. Inleiding

Dit functioneel ontwerp beschrijft **hoe** het systeem zich gedraagt vanuit het perspectief van de gebruiker. Het documenteert de schermen, navigatiestromen, use cases en procesflows van UC3 – het chatsysteem. ```

```
│                    ShowcaseWebsite                          │

│  │ Profiel  │   │ Contact  │   │   Chatsysteem        │   │

│  └──────────┘   └──────────┘   └──────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
         ▲                               ▲
    Bezoeker                    Ingelogde User / Admin
```

UC3 bestaat uit drie subsystemen:
1. **Authenticatiemodule** – registratie, login, 2FA, logout
2. **Chatmodule** – realtime berichten via SignalR + Vue 3
3. **Adminmodule** – berichtbeheer (alleen Admin-rol)

## 3. Actoren

| Actor | Omschrijving |
|-------|-------------|
| **Bezoeker** | Niet-ingelogde gebruiker; kan alleen publieke pagina's zien |
| **User** | Geregistreerde en ingelogde gebruiker; kan chatten en 2FA beheren |
| **Admin** | Speciale rol met toegang tot admin-dashboard; kan ook chatten |
| **Systeem** | De applicatie zelf (automatische acties: seeding, auto-reconnect) |

## 5. Use Case Beschrijvingen

### UC-A1: Registreren

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-A1: Registreren |
| **Actor** | Bezoeker |
| **Precondities** | Gebruiker heeft nog geen account |
| **Postcondities** | Account aangemaakt, rol "User" toegewezen, gebruiker ingelogd, redirect naar Chat |

**Normaal verloop:**
1. Bezoeker navigeert naar `/Account/Register`
2. Bezoeker vult in: e-mailadres, weergavenaam, wachtwoord, wachtwoord (bevestiging)
3. Systeem valideert invoer (vereiste velden, wachtwoordbeleid, uniek e-mailadres)
4. Systeem maakt account aan in database
5. Systeem wijst rol "User" toe
6. Systeem logt gebruiker automatisch in
7. Systeem redirect naar `/Chat`

**Alternatief verloop – Validatiefout:**
3a. Validatie mislukt → foutmeldingen getoond bij velden → gebruik kiest opnieuw

**Alternatief verloop – Duplicaat e-mail:**
3b. E-mailadres al in gebruik → foutmelding "E-mailadres is al in gebruik"

---

### UC-A2: Inloggen

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-A2: Inloggen |
| **Actor** | Bezoeker / uitgelogde User |
| **Precondities** | Account bestaat |
| **Postcondities** | Gebruiker is ingelogd (of 2FA-verificatie gestart) |

**Normaal verloop:**
1. Bezoeker navigeert naar `/Account/Login`
2. Bezoeker vult e-mailadres en wachtwoord in
3. Systeem verifieert credentials
4a. **Zonder 2FA:** Gebruiker is direct ingelogd → redirect naar `/Chat`
4b. **Met 2FA:** Systeem redirect naar Verify2FA-pagina

**Alternatief – Fout wachtwoord:**
3a. Verkeerd wachtwoord → "Ongeldige inloggegevens" (bewust vaag voor beveiliging)
3b. Na 5e poging → account vergrendeld 15 minuten → foutmelding getoond

**Alternatief – Vergrendeld account:**
4c. Vergrendeld → "Account tijdelijk vergrendeld, probeer later" → geen redirect

### UC-A3: Uitloggen

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-A3: Uitloggen |
| **Actor** | User / Admin |
| **Precondities** | Gebruiker is ingelogd |
| **Postcondities** | Sessie-cookie verwijderd, redirect naar startpagina |

**Normaal verloop:**
1. Gebruiker klikt op "Uitloggen" in navigatiebalk

3. Systeem verwijdert sessie-cookie

### UC-A4: 2FA Activeren

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-A4: 2FA Activeren |
| **Actor** | User / Admin (ingelogd) |
| **Precondities** | Gebruiker is ingelogd; 2FA nog niet actief |
| **Postcondities** | 2FA ingeschakeld voor het account |

**Normaal verloop:**
1. Gebruiker navigeert naar `/Account/Enable2FA`
2. Systeem genereert TOTP-geheimsleutel (Base32)
3. Systeem genereert QR-code (PNG, `otpauth://totp/...`-URI)
4. Pagina toont: QR-code, handmatige sleutel (geformatteerd in blokken van 4)
5. Gebruiker scant QR-code in authenticator-app (Google Authenticator, Aegis, etc.)
6. Gebruiker voert 6-cijferige code in en klikt "Activeren"
7. Systeem verifieert code → 2FA ingeschakeld → bevestigingsbericht

---

### UC-A5: 2FA Verifiëren

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-A5: 2FA Verifiëren |
| **Actor** | User / Admin (halverwege login) |
| **Precondities** | Wachtwoord correct ingevoerd; 2FA actief op account |
| **Postcondities** | Gebruiker volledig ingelogd |

**Normaal verloop:**
1. Na correcte wachtwoordinvoer → systeem redirect naar `/Account/Verify2FA`
2. Gebruiker opent authenticator-app, ziet actuele 6-cijferige code
3. Gebruiker voert code in (spaties/koppeltekens worden automatisch verwijderd)
4. Systeem verifieert code → gebruiker ingelogd → redirect naar `/Chat`

**Alternatief – Ongeldige code:**
4a. Verkeerde code → "Ongeldige verificatiecode"

### UC-C1: Chatbericht Sturen

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-C1: Chatbericht Sturen |
| **Actor** | User / Admin (ingelogd, verbonden) |
| **Precondities** | Gebruiker is ingelogd en Vue-component is verbonden met ChatHub |
| **Postcondities** | Bericht opgeslagen in DB, ontvangen door alle verbonden clients |

**Normaal verloop:**
1. Gebruiker typt bericht in invoerveld (max. 1000 tekens, `maxlength` in HTML + servervalidatie)
2. Gebruiker klikt "Verzenden" of drukt Enter
3. Vue-component roept `connection.invoke('SendMessage', content)` aan
4. ChatHub ontvangt het bericht; haalt sender-identiteit op via `Context.UserIdentifier`
5. Hub roept `MessageService.SendMessageAsync(senderId, content)` aan
6. MessageService valideert, trimt en slaat op in database
7. Hub roept `Clients.All.SendAsync('ReceiveMessage', ...)` aan
8. Alle verbonden Vue-clients ontvangen het bericht en voegen het toe aan de lijst
9. Scherm scrollt automatisch naar beneden

**Alternatief – Leeg bericht:**
2a. Verzendknop is uitgeschakeld als invoerveld leeg of alleen spaties
3a. Eventueel servervalidatie: ArgumentException → ReceiveError terug naar sender

**Alternatief – Bericht te lang:**
2b. HTML maxlength=1000 blokkeert verdere invoer
3b. Servervalidatie gooit ArgumentException → foutmelding 4 seconden zichtbaar

**Alternatief – Verbinding verbroken:**
2c. Verzendknop is uitgeschakeld als `connected = false`

### UC-C2: Chatbericht Ontvangen

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-C2: Chatbericht Ontvangen |
| **Actor** | User / Admin (ingelogd, verbonden) |
| **Precondities** | Gebruiker is verbonden met ChatHub |
| **Postcondities** | Bericht zichtbaar in berichtenlijst |

**Normaal verloop:**
1. Een andere gebruiker stuurt een bericht (UC-C1)
2. Hub stuurt `ReceiveMessage(sender, content, time)` naar alle clients
3. Vue-component voegt `{sender, content, time}` toe aan `messages` array
4. Eigen berichten krijgen een andere achtergrondkleur (Bootstrap `list-group-item-primary`)
5. Berichtcontainer scrollt automatisch naar nieuwste bericht

### UC-C3: Berichtenhistorie Bekijken

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-C3: Berichtenhistorie Bekijken |
| **Actor** | User / Admin |
| **Precondities** | Gebruiker navigeert naar `/Chat` |
| **Postcondities** | Scherm toont bestaande berichten |

**Normaal verloop:**

### UC-D1: Alle Berichten Inzien (Admin)

| Veld | Inhoud |
|------|--------|
| **Naam** | UC-D1: Alle Berichten Inzien |
| **Actor** | Admin |
| **Precondities** | Gebruiker heeft Admin-rol |
| **Postcondities** | Alle berichten (nieuwste eerst) zichtbaar in dashboard |

| Veld | Inhoud |

| **Actor** | Admin |

## 6. Navigatiestructuur

```
/                   → Home/Index (publiek)
/Account/Register   → Registratieformulier (publiek)
/Account/Login      → Loginformulier (publiek)
/Account/Logout     → POST only, redirect naar /
/Account/Enable2FA  → 2FA activeren (alleen ingelogd)
/Account/Verify2FA  → 2FA verificatie (halverwege login)
/Chat               → Chatpagina (alleen ingelogd)
/Admin              → Admin-dashboard (alleen Admin-rol)
/Admin/FilterByUser/{userId} → Gefilterde berichten (alleen Admin-rol)
```

**Redirect-regels:**
- Niet-ingelogde bezoeker probeert `/Chat` → redirect naar `/Account/Login?returnUrl=/Chat`
- Niet-admin probeert `/Admin` → redirect naar `/Account/Login`
- Na succesvolle login → redirect naar `returnUrl` of `/Chat`

## 8. Processtromen

### 8.1 Loginflow (inclusief 2FA)

```
Start
  │
  ▼
Invoer e-mail + wachtwoord
  │
  ▼
Credentials correct?
  ├── Nee → Foutmelding
  │         Pogingteller +1
  │         5 pogingen? → Account vergrendeld 15 min
  │
  └── Ja → 2FA ingeschakeld?
              ├── Nee → Sessie aanmaken → Redirect /Chat
              └── Ja  → Redirect /Verify2FA
                          │
                          ▼
                       Code invoeren
                          │
                          ▼
                       Code geldig?
                          ├── Nee → Foutmelding
                          └── Ja  → Sessie aanmaken → Redirect /Chat
```

### 8.2 Bericht-stuurflow (SignalR)

```
Browser (Vue)                     Server (ChatHub)              Database
    │                                    │                          │
    │── invoke('SendMessage', text) ────►│                          │
    │                                    │── SendMessageAsync() ───►│
    │                                    │◄─ Message (met Sender) ──│
    │                                    │                          │
    │◄── ReceiveMessage() (alle clients) │                          │
    │    (sender, content, time)         │                          │
    │                                    │                          │
    │ Bericht toegevoegd aan lijst       │                          │
    │ Auto-scroll naar beneden           │                          │
```

### 8.3 Autorisatieflow

```
HTTP-verzoek
    │
    ▼
UseAuthentication()  ← Laad sessieprincipal uit cookie
    │
    ▼
UseAuthorization()
    │
    ▼
[Authorize] aanwezig?
    ├── Nee → Verzoek doorgelaten (publieke pagina)
    └── Ja  → Gebruiker ingelogd?
                ├── Nee → Redirect /Account/Login?returnUrl=...
                └── Ja  → Rol-eis aanwezig?
                            ├── Nee → Toegang verleend
                            └── Ja  → Gebruiker heeft rol?
                                        ├── Nee → 403 / Redirect Login
                                        └── Ja  → Toegang verleend
```

## 11. Business Rules

| BR-ID | Regel |
|-------|-------|
| BR01 | Een bericht mag maximaal 1000 tekens bevatten |
| BR02 | Een leeg bericht (inclusief alleen spaties) wordt geweigerd |
| BR03 | De afzender-identiteit wordt altijd server-side bepaald (nooit via client-invoer) |
| BR04 | Elke nieuwe gebruiker krijgt de rol "User"; Admin-rol wordt alleen via seeding of beheerdersactie toegewezen |
| BR05 | Account wordt vergrendeld na 5 opeenvolgende mislukte inlogpogingen voor 15 minuten |
| BR06 | Sessie verloopt na 8 uur inactiviteit |
| BR07 | Berichten worden opgeslagen in UTC-tijdzone, weergegeven in lokale tijd (HH:mm) |
| BR08 | Maximaal 100 berichten worden geladen bij opening van de chatpagina |
